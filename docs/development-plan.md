# 心电图学习平台一期开发计划（V1）

## 1. 计划目标

本计划用于把需求文档与架构文档转化为具体的开发步骤、阶段目标、交付物与实施顺序，确保项目可以按 MVP 范围快速落地。

## 2. 一期目标定义

一期 MVP 的目标是交付一个可演示、可持续迭代的学习平台，满足以下结果：

- 管理员可以登录后台，管理心电图案例、分类、标签与测验题
- 用户可以在 Web 与 Android 上浏览案例、学习内容并完成测验
- 系统可以记录收藏、学习进度、错题与答题结果
- 后端、数据库、对象存储与 Web 托管具备 Docker 化运行能力
- 项目具备统一 UI 库与基础设计规范

## 3. 项目实施原则

- 先打通闭环，再补细节
- 先做内容管理，再做学习体验增强
- 先做基础设计系统，再做页面扩展
- 先保证结构化数据可维护，再做复杂统计与智能化能力

## 6. 建议开发顺序

推荐采用下面的执行顺序，这样最容易打通业务闭环。

1. 先完成工程初始化与目录搭建
2. 先完成设计 Token 与基础组件
3. 先完成后端案例管理主链路
4. 再完成管理端案例录入与发布
5. 再完成用户端案例浏览与详情
6. 最后完成测验、收藏、错题本与学习记录

## 8. 建议迭代节奏

如果按小步快跑推进，建议节奏如下：

- 第 1 周：工程初始化 + 架构骨架 + Docker 基础
- 第 2 周：UI 库基础 + 后端鉴权与案例模型
- 第 3 周：管理端案例管理闭环
- 第 4 周：用户端案例浏览与详情
- 第 5 周：测验、收藏、错题本、学习记录
- 第 6 周：联调、测试、部署优化

说明：这是偏顺利情况下的 MVP 节奏，实际可按人力再拆分。

## 9. 风险与应对

### 风险 1：Flutter Web 后台体验细节较多
- 应对：后台优先保证可用性与信息密度，不追求复杂交互动画

### 风险 2：图片管理与展示复杂度偏高
- 应对：一期先做标准图片上传与浏览，延后高级标注能力

### 风险 3：医学内容字段很多，表单容易过重
- 应对：后台编辑页分区展示，支持草稿保存

### 风险 4：Android 与 Web 共用代码时布局差异明显
- 应对：在案例详情页和测验页提前做响应式抽象

### 风险 5：后期 UI 与 Figma 设计系统脱节
- 应对：优先把 `ecg_ui` 作为代码真源，并保持 Figma Token 对齐

## 10. 一期完成标准

满足以下条件，可认为一期 MVP 达成：

- 管理员可维护完整案例数据
- 用户可在 Web 与 Android 上完成学习和测验
- 关键数据可持久化保存
- 核心接口与页面稳定可演示
- Docker 化开发与演示环境可复现
- 文档、目录结构与设计系统基础已建立

## 11. 下一步建议

下一阶段建议直接进入“项目骨架搭建”：

1. 创建 Monorepo 目录
2. 初始化 Flutter 用户端工程
3. 初始化 Flutter 管理端工程
4. 初始化 FastAPI 工程
5. 初始化 Docker Compose
6. 创建 `ecg_ui` 与基础主题

## 12. 结论

当前最合理的推进方式不是先做所有页面，而是优先完成三条主线：

- 工程主线：把仓库、服务、环境和目录搭起来
- 内容主线：把案例管理闭环做通
- 学习主线：把用户学习和测验闭环做通

等这三条主线稳定之后，再去补统计、审核流、设计细节与二期能力，项目推进会更稳。
