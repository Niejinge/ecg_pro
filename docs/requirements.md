# 心电图学习平台需求文档（V1）

## 1. 项目概述

### 1.1 项目名称
- 暂定名称：`ECG Pro` / `心电图学习平台`

### 1.2 项目背景
本项目面向心电图学习场景，提供标准化案例学习、结构化诊断理解、随堂测验与复习能力，帮助学习者从“看图”逐步过渡到“理解诊断逻辑与临床处理思路”。

### 1.3 项目目标
- 搭建一个同时包含管理端与用户端的心电图学习平台。
- 管理端支持标准化维护心电图案例、题库与学习内容。
- 用户端支持按主题学习案例、完成测验、查看解析与追踪学习进度。
- 首期优先支持 `Web` 与 `Android`。
- 前端采用 Flutter，后端采用 FastAPI，整体通过 Docker 进行部署和维护。
- 建立统一的 UI 组件库与设计规范，风格清新、简洁、接近 Apple 风格。

### 1.4 一期定位
一期以“可用、可扩展、便于持续补充案例”为核心目标，先实现：
- 后台内容管理
- 用户学习与测验
- Web 与 Android 端可访问
- 基础学习数据追踪

以下内容建议放到二期：
- 社区讨论
- AI 自动判读
- 视频课程
- 医院/机构多租户
- 复杂统计报表

## 2. 用户角色

### 2.1 管理端角色
- 超级管理员：系统配置、管理员账号管理、全部数据权限。
- 内容管理员：案例录入、编辑、发布、题目录入、分类维护。
- 审核管理员：审核案例质量、控制发布状态。

说明：一期可先简化为 `超级管理员 + 内容管理员` 两类角色。

### 2.2 用户端角色
- 游客：浏览公开案例列表与部分详情，不记录学习数据。
- 注册用户：完整学习、参与测验、收藏、查看错题与进度。

说明：一期建议支持游客浏览、登录后记录学习进度。

## 3. 产品范围

### 3.1 管理端功能范围
- 管理员登录
- 仪表盘概览
- 心电图案例管理

- 分类/标签管理
- 测验题目管理
- 案例发布与下线

### 3.2 用户端功能范围
- 首页推荐
- 案例库浏览
- 分类筛选与搜索

- 收藏与错题本
- 学习记录与进度
- 个人中心

## 4. 核心业务流程

### 4.1 管理端案例发布流程
1. 管理员创建案例
2. 上传心电图图片
3. 填写结构化案例信息
4. 绑定分类、标签、难度、风险等级
5. 添加配套测验题目
6. 预览案例
7. 保存为草稿或提交发布
8. 发布后在用户端展示

### 4.2 用户端学习流程
1. 用户进入首页或案例库
2. 按分类、难度、标签筛选案例
3. 查看案例详情与诊断解析
4. 完成案例相关测验
5. 查看答案解析与推荐复习内容
6. 系统记录学习进度、错题与掌握情况

## 5. 心电图案例内容规范

每个案例建议采用“结构化医学信息 + 学习辅助信息”双层设计。

### 5.1 基础信息
- 案例标题
- 病例编号
- 案例封面图
- 心电图原图
- 心电图附加图（可多张）
- 分类
- 子分类
- 标签
- 难度等级
- 是否推荐案例
- 发布状态

### 5.2 医学内容字段
- 诊断名称
- 英文诊断名（可选）
- 节律类型
- 心率范围
- 电轴情况
- P 波特征
- PR 间期
- QRS 波群特征
- ST-T 改变
- QT/QTc 情况
- 关键导联表现
- 临床意义
- 鉴别诊断
- 风险等级
- 危急值提示
- 治疗原则
- 紧急处理建议
- 后续检查建议
- 相关禁忌或注意事项

### 5.3 病例背景字段
- 患者性别（可匿名）
- 患者年龄段
- 主诉
- 简要病史
- 既往史/诱因
- 相关症状
- 临床场景说明

说明：如不方便放真实病例信息，可使用脱敏后的教学案例。

### 5.4 学习辅助字段
- 详细描述
- 分步判读过程
- 判读要点
- 常见误区
- 记忆技巧
- 学习总结
- 推荐前置知识
- 推荐关联案例

### 5.5 内容治理字段
- 创建人
- 审核人
- 创建时间
- 更新时间
- 发布时间
- 版本号
- 备注

### 5.6 风险等级建议标准
- `低风险`：通常不属于急危重处理场景，适合基础识别学习。
- `中风险`：需要进一步评估或尽快处理。
- `高风险`：可能提示严重心律失常或明显缺血改变。
- `危急`：需立即识别并快速处理的高危心电图表现。

## 6. 测验功能需求

### 6.1 题型范围
一期建议先支持：
- 单选题
- 多选题
- 判断题

二期可扩展：
- 填空题
- 导联定位题

### 6.2 测验模式
- 案例学习后即时测验
- 按分类专项测验
- 按难度随机练习
- 错题重做

### 6.3 测验能力要求

- 支持正确答案与错误原因说明
- 支持记录答题结果
- 支持统计正确率
- 支持记录最近作答时间
- 支持生成错题本
- 支持按分类查看薄弱点

## 9. 平台与技术要求

### 9.1 前端要求
- 使用 Flutter 开发
- 首期支持 `Web` 与 `Android`
- 用户端与管理端都基于 Flutter 实现
- 支持响应式布局

建议方案：
- `用户端`：Flutter Web + Flutter Android
- `管理端`：Flutter Web 优先

说明：管理端通常不要求 Android 形态，一期建议先以 Web 后台为主。

### 9.2 后端要求
- 使用 FastAPI
- 提供 RESTful API
- 自动生成 OpenAPI 文档
- 支持用户认证、权限控制、内容管理、学习记录管理

### 9.3 存储建议
- 数据库：建议使用 `PostgreSQL`
- 图片存储：开发期可用本地卷，后续建议接 `MinIO` 或对象存储
- 缓存：二期可加入 `Redis`

### 9.4 部署要求
- 所有服务通过 Docker 维护
- 开发阶段建议使用 `docker compose`
- 至少包含以下容器：
  - Flutter Web 构建/静态托管
  - FastAPI 服务
  - PostgreSQL
  - MinIO（可选，但建议纳入）
  - Nginx（可选，用于统一网关与静态资源分发）

## 10. UI/UX 设计要求

### 10.1 风格要求
- 整体风格清新、简单、轻量
- 接近 Apple 风格，但避免完全照搬
- 强调留白、层级、圆角、柔和阴影与浅色系
- 重点突出“学习效率”和“信息可读性”

### 10.2 视觉关键词
- 清爽
- 安静
- 可信
- 专业
- 易读

### 10.3 设计规范要求
- 建立统一颜色体系
- 建立统一字体层级
- 建立统一间距体系
- 建立统一圆角与阴影规范
- 建立统一按钮、卡片、标签、输入框、弹窗、导航等组件

### 10.4 页面体验要求
- 图片查看要清晰，支持点击放大
- 案例详情阅读节奏清楚
- 测验反馈直观
- 移动端单手操作友好
- Web 端信息布局清晰不过密

### 10.5 Figma 与设计系统要求
- 使用 Figma 维护设计稿与组件库
- 建立基础 Design Tokens：
  - 颜色
  - 字体
  - 间距
  - 圆角
  - 阴影
  - 组件状态
- 后续可通过 Figma MCP 辅助生成统一 UI 库与页面草图

## 11. 非功能性需求

### 11.1 性能
- Web 首屏加载应尽量控制在 3 秒内
- 图片需要压缩与缩略图能力
- 案例列表支持分页加载

### 11.2 安全
- 后台接口必须鉴权
- 图片上传需校验类型与大小
- 敏感操作需记录日志
- 医学案例需支持脱敏

### 11.3 可维护性
- 前后端模块边界清晰
- 接口文档自动化
- 支持 `.env` 配置
- 支持 Docker 本地一键启动

### 11.4 医学内容规范
- 平台定位为教学用途
- 页面需注明“仅供学习参考，不替代临床诊疗”
- 案例内容应支持审核流程

## 14. 待确认问题

以下问题建议在进入开发计划前确认：
- 管理端是否只做 Web，不做 Android？
- 用户端是否需要短信/邮箱注册登录，还是先做简单账号登录？
- 心电图案例图片来源是否已有现成素材库？
- 是否需要管理员审核发布流程，还是内容管理员可直接发布？
- 一期是否需要游客访问，还是必须登录后才能学习？
- 是否需要支持中英文双语内容？

## 15. 结论

本项目的一期目标可以定义为：

“构建一个基于 Flutter + FastAPI + Docker 的心电图学习平台，包含管理端内容维护与用户端学习测验能力，首期交付 Web 与 Android，并同步建立统一的 Apple 风格轻量设计系统。”

后续开发建议顺序为：
1. 明确一期 MVP 范围
2. 输出系统架构与目录结构
3. 设计数据库与接口草案
4. 设计 Figma 组件库与页面原型
5. 按模块开始开发
